### "이걸 왜 만들었지?"

> 사실 처음 Claude Artifacts 런칭 소식을 들었을 때 회의적이었다.

'그냥 몇 마디로 이미지를 만들어주는 기능' 정도로 생각했다. 이미 Figma도 있고, 코드를 직접 작성하는게 더 빠르다고 생각했다. 스타트업 개발자로서 사이드 프로젝트도 여럿 진행하는 입장에서, 새로운 툴을 익히는 시간이 아깝게 느껴졌다.

하지만 **최근 Artifacts를 처음 '제대로' 활용해보면서, 완전히 생각이 바뀌었다.** 그동안 몇 가지 개선이 있었는지, 처음 생각했던 것과는 차원이 다른 경험이었다.

<br>

### 본질적으로 '퍼블리싱'까지 해주는 기능?

Artifacts를 사용하면서 가장 인상깊었던 건, **실제로 작동하는 애플리케이션이 바로 화면에 나타난다**는 점이었다.

코드로 작업하는 게 아니라, 대화만으로 결과물이 실시간으로 렌더링된다. 그런데 신기한 건 소스코드를 확인해보면 리액트(React)로 작성되어 있다는 것. 이게 처음엔 혼란스러웠다.

```
"이게 무슨 상황이지?
코드는 리액트인데, 내가 코딩한 건 아니고,
그렇다고 노코드도 아니고..."
```

**앤트로픽은 이걸 왜 만들었을까?** 곰곰이 생각해보니 답이 보였다.

Artifacts는 단순히 "코드를 대신 작성해주는 도구"가 아니다. **아이디어를 즉각적으로 시각화하고, 검증하고, 반복하는 과정을 극적으로 단축시켜주는 도구**다.

- 개발자에게는: 프로토타입을 빠르게 만들고 테스트할 수 있는 환경
- 비개발자에게는: 기술적 장벽 없이 아이디어를 구현할 수 있는 플랫폼
- 모두에게: 커뮤니케이션 비용을 줄이고, 빠른 실험을 가능하게 하는 도구

<br>

### 이제 프로토타입은 무조건 Artifacts로 만든다

결론부터 말하자면, **이제 프로토타입은 Artifacts로 만든다. 피그마는 안 쓴다.**

스타트업에서 일하면서, 그리고 여러 사이드 프로젝트를 진행하면서 느낀 가장 큰 병목은 "아이디어 → 프로토타입"으로 가는 과정이었다.

**기존 워크플로우:**
1. 피그마로 디자인 스케치
2. 개발 환경 세팅
3. 컴포넌트 작성
4. Vercel 등에 배포
5. 피드백 받고 다시 1번부터...

**Artifacts 워크플로우:**
1. Claude에게 설명
2. 바로 확인하고 수정 요청
3. 완성

시간으로 따지면 **몇 시간이 몇 분으로** 줄어들었다. 더 중요한 건 **심리적 장벽**이 사라졌다는 것. "개발 환경 세팅하고, 보일러플레이트 코드 작성하고..." 이런 생각만 해도 귀찮아서 미루던 아이디어들을 이제는 바로 시도해볼 수 있다.

<br>

### Lovable + 피그마 + Vercel을 한 번에

Artifacts를 사용하면서 **"Lovable, 피그마, Vercel을 모두 한 번에 사용하는 것 같다"**는 느낌을 받았다.

- **Lovable처럼**: AI가 코드를 작성해주지만
- **피그마처럼**: 즉각적인 시각적 피드백을 받을 수 있고
- **Vercel처럼**: 바로 공유하고 테스트할 수 있는 환경

물론 한계도 있다. DB를 직접 연동하거나, 복잡한 백엔드 로직을 구현하긴 어렵다. 하지만 **프로토타입 단계에서 필요한 건 거의 다 가능하다**는 게 핵심이다.

실제로 최근 진행 중인 사이드 프로젝트에서:
- 랜딩 페이지 프로토타입: 10분
- 대시보드 UI 목업: 15분
- 사용자 플로우 시뮬레이션: 20분

이 정도 속도로 검증할 수 있었다. Mock 데이터로 충분히 작동하는 프로토타입을 만들 수 있었고, 팀원들에게 공유해서 즉각적인 피드백을 받을 수 있었다.

<br>

### 스타트업 개발자가 Artifacts를 활용하는 법

스타트업에서 일하고 사이드 프로젝트도 진행하는 개발자로서, Artifacts를 언제 활용하면 좋을지 정리해봤다.

**1. 아이디어 검증 단계**
- 고객에게 보여줄 간단한 목업이 필요할 때
- "이렇게 작동하면 어떨까?" 시뮬레이션이 필요할 때
- MVP를 만들기 전 컨셉 프루프가 필요할 때

**2. 팀 커뮤니케이션**
- 디자이너와 인터랙션 논의할 때
- 기획자에게 기능 설명할 때
- 투자자나 고객에게 데모를 보여줄 때

**3. 빠른 실험**
- 새로운 라이브러리나 UI 패턴 시도
- A/B 테스트용 페이지 여러 버전 제작
- 마케팅 랜딩 페이지 빠르게 제작

**언제 사용하지 말아야 할까?**
- 프로덕션 코드가 필요할 때
- 복잡한 상태 관리가 필요할 때
- 백엔드 통합이 필수적일 때
- 성능 최적화가 중요할 때

핵심은 **"검증"과 "커뮤니케이션"**이다. 완성된 제품을 만드는 도구가 아니라, 올바른 제품을 만들기 위한 방향을 빠르게 찾는 도구로 활용하는 것이다.

<br>

### 워크플로우의 변화

Artifacts를 도입하면서 프로토타이핑 워크플로우가 완전히 바뀌었다.

**Before:**
```
아이디어 → 피그마 디자인 → 개발 환경 세팅 → 구현 → 배포 → 피드백
(소요 시간: 몇 시간 ~ 며칠)
```

**After:**
```
아이디어 → Artifacts로 구현 & 검증 → (필요시) 실제 개발
(소요 시간: 몇 분 ~ 몇십 분)
```

특히 **"실패하는 아이디어를 빨리 버릴 수 있게 되었다"**는 게 가장 큰 변화다. 예전에는 이미 시간을 많이 투자한 아이디어는 애착이 생겨서 객관적 판단이 어려웠다. 하지만 이제는 10분 만에 만들어서 테스트하고, 안 되면 바로 다음 아이디어로 넘어갈 수 있다.

<br>

### 결론: 앤트로픽의 의도를 이제야 이해했다

처음 "이걸 왜 만들었지?"라고 생각했던 게 부끄러울 정도다.

앤트로픽은 단순히 "코드를 작성해주는 AI"를 만든 게 아니었다. **아이디어와 실행 사이의 간극을 없애는 도구**를 만든 것이다.

개발자에게 가장 중요한 건 코드를 작성하는 능력이 아니라, 올바른 것을 만드는 능력이다. Artifacts는 바로 그 "올바른 것을 찾는 과정"을 극적으로 개선해준다.

앞으로 모든 프로토타입은 Artifacts로 시작할 것이다. 그리고 검증된 아이디어만 실제 개발로 넘어갈 것이다. 이게 2025년 스타트업 개발자의 새로운 워크플로우가 아닐까.

<br>
